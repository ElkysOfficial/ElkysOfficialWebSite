## Parte I — Inconsistências de Layout e Design System

### 1. Portal Admin

Conformidade ao DS estimada em ~65%. Principais desvios em ordem de impacto:

#### 1.1 Cores hardcoded — crítico

- **`Tasks.tsx:102-109`** — constantes `CATEGORIES` com `bg-blue-500`, `bg-cyan-500`, `bg-emerald-500`, `bg-amber-500`, `bg-pink-500`, `bg-purple-500`, `bg-slate-400` (8 cores Tailwind puras).
- **`Tasks.tsx:118-140`** — `PRIORITIES` com `bg-red-500`, `bg-orange-500`, `bg-yellow-500`, `bg-green-500` e variações `/15`, `/30` (12+ instâncias).
- **`Tasks.tsx:339`** — badge inline com `bg-blue-500/10 text-blue-600 dark:text-blue-400`.
- **`Tasks.tsx:763`** — notice de input com 3 cores hardcoded.
- **`Tasks.tsx:298`** — fallback `bg-slate-400` em ternário.

**Impacto:** ignoram o sistema de tokens — sem garantia de coerência dark mode e sem controle central. Solução: criar tokens `--elk-task-category-*` e `--elk-task-priority-*` ou mapear para `primary|secondary|accent|success|warning|destructive`.

#### 1.2 Inputs e botões HTML crus

- **`ChangePassword.tsx:117-124, 193-207`** — `<input>` puro com classes inline (devia usar `<Input>` + `<Field>` + `<ErrorText>`).
- **`ChangePassword.tsx:126-133, 209-216`** — `<button>` custom para toggle de senha (devia ser `<Button variant="ghost" size="icon">` com `aria-label`).
- **`Finance.tsx:568, 610, 626, 730`** — `<input>` cru com `min-h-[44px]` duplicando o padrão do DS.
- Inputs HTML crus detectados em **10 arquivos admin**: TeamCreate, Tasks, ProposalDetail, ProjectCreate, Leads, ClientCreate, ProjectDetail, LeadDetail, ClientDetail, BillingAutomation.

#### 1.3 Cards com styling inconsistente

- `<Card className="border-border/70 bg-card/92">` (ClientDetail)
- `<Card className="rounded-2xl border-border/80 bg-card/95">` (Finance)
- `<Card className="border-primary/30 bg-card/95">` (BillingAutomation) — **viola o DS**: bordas não devem usar `border-primary`.

**Correção:** `BillingAutomation.tsx:346, 501` → `border-border/70`.

#### 1.4 Borders com variações sem critério

`border-border/50`, `/60`, `/70`, `/80` usados intercambiavelmente. Sugestão: introduzir tokens `--elk-border-subtle` (/40), `--elk-border-medium` (/60), `--elk-border-strong` (/80) em `_tokens.scss`.

#### 1.5 Padding de card inconsistente

`p-3` (ChangePassword:92), `p-4` (Tasks, Expenses), `p-5` (Finance). Consolidar em `p-4 sm:p-5 lg:p-6`.

#### 1.6 Empty states e tabelas — OK

`AdminEmptyState` e HTML `<table>` com `border-border/60 bg-muted/30` estão coerentes.

### 2. Portal Cliente

Divergências principais são com o **portal admin** (dados iguais, UI diferente):

#### 2.1 Spacing dos KPI tiles diverge do admin

- **Cliente** `Finance.tsx:170`: `gap-2 min-[400px]:grid-cols-2 sm:gap-3 xl:grid-cols-3`.
- **Admin** `Finance.tsx:539`: `gap-3 min-[400px]:grid-cols-2 sm:gap-4 xl:grid-cols-3`.

Cliente respira 4px a menos em cada breakpoint. §14.1 já define o canônico `gap-3 sm:gap-4` — cliente deve subir.

#### 2.2 Tons de fundo em listas divergem

- Cliente: `bg-background/60` (linhas de projeto, cobrança) — visual mais "vazado".
- Admin: `bg-muted/25` ou `bg-card` — visual mais firme.

Impacto: mesma entidade renderizada com peso visual diferente nos dois portais. Padronizar em `bg-card` (linhas destacadas) + `bg-muted/30` (linhas secundárias).

#### 2.3 Support tickets com StatusBadge ad-hoc

- **`client/Support.tsx:42-50`** — `STATUS_CONFIG` manual com `bg-warning/10 text-warning` etc., em vez de usar `StatusBadge` ou `TICKET_STATUS_META` já centralizados (§14.2).

#### 2.4 Empty states "adminizados"

Todas as páginas cliente importam `AdminEmptyState`. Funciona, mas o tom cliente merece copy mais amigável. Sugestão: `ClientEmptyState` wrapper com copy default mais próximo ("Você ainda não tem projetos ativos" vs "Nenhum registro").

#### 2.5 Loading states mistos

- `Overview.tsx:58-76` — skeleton inline `h-[76px] animate-pulse rounded-2xl ...`.
- Finance/Contracts — `<PortalLoading>` spinner.

Definir regra: skeleton em listagens/Overview (mostra estrutura), spinner só em transições de rota.

#### 2.6 Typography eyebrow divergente

- Padrão: `text-[11px] font-semibold uppercase tracking-[0.08em]` (Finance).
- Divergência: **`ProposalView.tsx:327`** usa `text-[10px]` — 1px a menos, quebra ritmo.

#### 2.7 Contraste fraco em toggle de senha

- **`ChangePassword.tsx:137, 223`** — `text-muted-foreground/60` é cinza demais. Remover `/60`.

### 3. Marketing / Landing

Conformidade ~85% — melhor que os portais. Achados:

#### 3.1 Cores hardcoded em stats do hero

- **`Hero.tsx:137-147`** — `text-[hsl(180,75%,60%)]` 3× (era para ser `text-accent`).
- **`Cases.tsx:382, 388, 392`** — mesmo padrão replicado.

#### 3.2 About CTA sem usar Button

- **`About.tsx:131`** — Link estilizado manualmente com `group-hover:bg-gradient-primary`. Usar `<Button variant="gradient">` ou `buttonVariants()`.

#### 3.3 Legal pages — listas manuais

- `PrivacyPolicy.tsx:103-122` — `list-none ml-6 space-y-1`. `@tailwindcss/typography` está proibido; solução atual é aceitável mas não escala.

#### 3.4 Restante — OK

- Section spacing `py-16 md:py-20 lg:py-24` consistente em About, Services, Testimonials, ContactForm, Footer, Cases, ComoTrabalhamos.
- Hierarquia H1/H2/H3 coerente.
- Hover states, Navigation transparent/scrolled, Footer, Login/NotFound — todos OK.

### 4. Formulários

#### 4.1 ContactForm não usa componentes DS

- **`ContactForm.tsx:150-230`** — usa `<label>` manual, `<p className="text-destructive">` manual para erro, asterisco `*` hardcoded nos labels.
- Deveria usar `<Field>`, `<Label required>`, `<ErrorText>`.

#### 4.2 Ausência de máscaras

- Nenhum form tem máscara de telefone, CEP, CPF ou CNPJ aplicada consistentemente. `lib/masks.ts` existe mas não é universalmente adotado.

#### 4.3 Restante — OK

- Login, AddContractLinkForm seguem Field/Label/Input/ErrorText corretamente.
- Mensagens zod em pt-BR padronizadas.
- Submit com `loading` + "Salvando...", `min-h-[44px]` respeitado.
- Estados `disabled:opacity-50` coerentes.
